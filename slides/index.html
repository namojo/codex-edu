03

왜 Codex인가

답변형이 아니라 작업형
📁파일 단위

코드와 문서를 함께 다루고, 산출물을 파일로 남깁니다.

🔧도구 실행

검색·셸·패치를 연결해 답변보다 실행 결과를 냅니다.

🔁작업 루프

Plan → Build → Verify 흐름과 수정 라운드에 적합합니다.

♻️재사용

반복 업무를 Skill·템플릿으로 축적해 확장합니다.

ChatGPT — 옆의 상담사
  • 단위는 채팅방, 결과는 답변·초안
  • 사람이 복사·붙여넣기로 옮김
Codex — 폴더 여는 실무 담당자
  • 단위는 프로젝트 폴더, 결과는 파일·검수
  • 폴더 안에서 직접 생성·수정, 규칙은 재사용
핵심더 똑똑한 답변보다 재현 가능한 작업 시스템을 만드는 것이 Codex 활용의 출발점입니다.
04

Codex를 쓰는 3가지 방법 · 설치와 로그인

①Codex CLI

터미널에서 codex 명령. 이 과정의 기본 — 폴더에서 직접 만들고 실행.

②ChatGPT 앱

브라우저에서 chatgpt.com/codex. 설치 없이 가볍게, "앱에서 열기" 실습.

③IDE 확장

VS Code 등 편집기 안. 익숙한 화면에서 코드를 보며 함께 작업.

npm install -g @openai/codex # macOS: brew install codex codex login # ChatGPT 계정 (Plus/Pro/Team) codex login status # 'logged in'이면 준비 완료
터미널이 처음이라면 Windows: 시작 → PowerShell 검색 → 마우스 오른쪽 → 관리자 권한으로 실행. macOS: ⌘+Space → 터미널 → Enter. 명령은 복사해 붙여넣고 Enter. 설치 직후 codex가 안 잡히면 창을 닫고 새로 엽니다.

일 잘 맡기는 프롬프트 6요소

길이보다 구조
01목표

무엇을 완성해야 하는가 — 한 문장으로.

02입력 자료

참고 파일·링크·기존 코드·디자인 방향.

03제약 조건

범위·형식·톤·기술 스택·제외할 것.

04완료 기준

Done의 정의와 검증 항목.

05산출물

어떤 파일·경로·형태로 남길지.

06수정 기준

다음 라운드에서 무엇을 어떻게 바꿀지.

TIP"좀 더 좋게"보다 번호와 기준이 있는 요청이 훨씬 잘 작동합니다. "수정 포인트 목록도 함께 제안해줘"를 붙이면 다음 라운드가 쉬워집니다.
07

잘 맞는 일 vs 구조화가 필요한 일

✓ Codex에 잘 맞는 업무
  • 반복 수정과 리팩터링
  • 파일·폴더 단위 작업
  • 초안 생성 후 반복 개선
  • 체크리스트 기반 검수 · 실습/자동화로 이어질 일
⚠ 구조화가 더 필요한 업무
  • 목표가 모호한 요청
  • 참고 자료 없는 디자인 요청
  • 우선순위가 섞인 큰 요구사항
  • 완료 기준 없는 검수 · 한 번에 다 끝내려는 요청
관점문제가 도구의 한계인 경우도 있지만, 요청 구조의 문제인 경우가 더 많습니다. 잘 맞는 일을 먼저 고르고, 어려운 일은 더 잘게 구조화하세요.

세 축 — Agent · Skill · Tool

+ 4대 관통 원리
AAgent

누가 · 역할 · 경계. 하나의 목표를 맡는 실행 주체.

SSkill

무엇을 · 어떻게. 반복 가능한 지식·절차의 묶음.

TTool

무엇으로 실행. 결정론적 코드에 위임하는 능력.

1What·Who·How 분리

스펙·에이전트·스킬을 한 프롬프트에 뭉치지 않는다.

2지식의 외부화

규칙·지식을 프롬프트가 아니라 파일로 꺼낸다.

3계약이 병렬·반복

형식을 먼저 정하면 나누고 반복할 수 있다.

4검증자 분리

만든 주체가 자기 결과를 최종 승인하지 않는다.

실습 1

제안서 PPT 만들기 — 작업 폴더를 맡긴다

시나리오반복 보고·자료조사를 줄이는 4주 파일럿 제안서. 공개 자료와 가상 시나리오만 사용합니다.
먼저 기준 문서부터 (PPT는 나중)
  • AGENTS.md — 작업 규칙·말투·검수·안전선
  • brand/DESIGN.md — 색·폰트·여백 기준
  • context/business-context.md — 청중·문제·메시지
  • context/deck-brief.md — 슬라이드 흐름·완료 기준

채팅방이 아니라 작업 폴더를 맡깁니다 — 설치 → 폴더 → 기준 문서 → Plan → 생성 → 검수.

핵심 텍스트는 이미지에 굽지 않는다 — 사람이 나중에 고칠 수 있어야 실무 파일.

검수는 생성과 분리 — 렌더 이미지로 글자 잘림·겹침·과장 표현을 먼저 잡는다.

앱 조작 권한은 좁게 — "이 파일만·이 앱만", 계정·결제 화면은 건드리지 않게.

실습 2

멀티스레드 뉴스레터 — 팀장과 담당자

팀장 스레드는 직접 일하지 않고 지시·보고 취합·진행 결정만. 담당자 스레드가 실제 리서치·작성.

나눌지 판단하는 기준 3
  • 병렬성 — 동시에 돌릴 수 있나
  • 역할 분리 — 판단 기준이 다른가
  • 재사용 — 다시 이어갈 일인가
가장 자주 막히는 곳완료는 저절로 알려지지 않습니다. 담당자는 output/에 저장하고 완료: 한 줄과 <이름>.done을 남깁니다.
실습 2

페이즈 게이트 — 완료를 확인한 뒤에만

Phase단계시작 조건
1리서치 3종 (병렬)바로 시작
2팩트체크3개 .done 후
3구성(목차)검증 통과 자료로만
4글쓰기목차 확정 후
5이미지 + 검수본문 확정 후
6패키징(HTML)글·이미지 둘 다
매주 반복이면긴 지시문을 매번 붙여넣지 말고 AGENTS.md로 저장하면 같은 규칙이 자동 적용됩니다.

세 가지 일하는 방식 — L1 · L2 · L3

trend-report 모듈
L1단일

하나의 스레드로 한 번에. 일회성·짧은 작업에 가장 빠릅니다.

L2멀티

역할을 나눠 병렬로. 조사·작성·검수를 분리할 때.

L3하네스

스레드 팀·게이트·규칙을 자동 구성. 반복되는 큰 흐름에.

재현성의 서사L1과 L3의 실행 명령은 둘 다 한 줄입니다 — 차이는 그 한 줄이 기대는 자산(파일)의 양입니다. 처음엔 싱글로 돌려 보고, 병목이 보일 때 나눕니다.

실무 — 하네스 엔지니어링

잘 쓰기 vs 빠뜨리지 않기
대화형 = 승인
  • Codex가 파일 수정·명령 전 승인을 묻습니다
  • 내용을 확인하고 허용 — 사람이 게이트
자동화 = 샌드박스
  • 사람이 없으니 승인 대신 샌드박스로 통제
  • --sandbox read-only / workspace-write
codex exec --output-schema codex review CI 게이트
관찰 포인트테스트가 통과해도 진단이 결함을 잡습니다 — 초록불 ≠ 정확. 그래서 검증은 테스트와 별도로 필요하고, 매주 같은 구조는 --output-schema 계약으로 고정합니다.

확장 — MCP Tool 직접 붙이기

세 번째 축
📖 Skill = 설명서
  • 무엇을·어떻게 하는지 지식과 절차
  • 모델이 읽고 따라 하는 문서
🧮 Tool = 계산기
  • 결정론적 코드가 실행하는 능력
  • 모델이 암산하면 틀리는 계산을 위임
언제 Tool인가오픈소스 MCP 서버로 나만의 Tool을 만들면 대화형·자동화 양쪽에서 같은 서버를 씁니다. 모델이 추측하면 안 되는 정확한 계산·조회는 Tool로 넘깁니다.

자가학습 지도 — 9모듈 커리큘럼

입문 · 실무 · 확장
단계모듈한 줄 목표
입문xlsx표준용어집을 파일로 외부화해 제각각인 보고서를 하나로 정규화
trend-report같은 보고서를 단일 → 멀티 → 하네스 3단계로 만들어 본다
proposal같은 지시서로 단일 프롬프트 vs 전문가 에이전트 팀을 A/B 비교
slide기준일 셀 하나로 전체가 갱신되는 수집→갱신→디자인 파이프라인
map스펙(What)을 쓰고 그것을 구현할 하네스(Who·How)를 설계
dashboard표준 스키마(계약)를 확정한 뒤 팬아웃으로 병렬 생성
실무track-a-engineering진단→패치→교차 리뷰→CI 게이트로 "초록불 ≠ 정확" 체험
track-b-ops코드 없이 반복 업무를 규칙·형식 계약으로 자동화
확장tool오픈소스 MCP 서버로 나만의 Tool을 만들어 양쪽에서 사용

전부 할 필요는 없습니다 — xlsx → trend-report → proposal 세 개만 먼저 해도 하네스의 핵심은 잡힙니다.
