# 단일 프롬프트 vs 하네스 — 과업지시서로 검증하는 A/B 실험

> 나라장터 과업지시서 1건(국가통합바이오빅데이터 EMR 연계 ETL 구축, 양산부산대학교병원)을
> 입력으로, **① 단일 프롬프트**와 **② 동봉된 전문가 에이전트 팀(하네스)**으로 각각
> 사업 제안서(docx)를 작성하고 결과를 비교하는 실험 교안입니다.
> 두 방식의 예시 산출물과, 하네스를 이루는 구성물(AGENTS.md·에이전트·스킬) 전부가
> 이 디렉토리에 들어 있습니다.

## 1. 실험 질문

**"배경지식이 없는 상태에서, 전문가 에이전트 구조는 무엇을 바꾸는가?"**

과업지시서에는 FHIR, KCD-8, 심평원 EDI 같은 단어가 이름만 등장합니다. 발주 도메인
(보건의료 데이터)을 모르는 상태에서 좋은 제안서를 쓸 수 있는가 — 이것이 실험의
핵심 조건입니다.

## 2. 준비와 디렉토리 구성

```bash
npm install -g @openai/codex   # 또는 brew install codex
codex login
git clone https://github.com/namojo/codex-edu.git
cd codex-edu/proposal
```

```
proposal/
├── README.md                        ← 본 교안
├── 과업지시서_데이터뱅크.hwpx        ← 입력 (나라장터 원문)
├── 제안서A_단일패스.docx             ← 트랙 A 예시 산출물 (3쪽)
├── 제안서B_하네스.docx               ← 트랙 B 예시 산출물 (8쪽)
├── artifacts/
│   └── 요구사항_인벤토리.md          ← 트랙 B의 중간 산출물 (차이의 근원)
├── AGENTS.md                        ← 하네스: 프로젝트 불변 규칙 (Codex가 자동으로 읽음)
├── .codex/
│   └── agents/                      ← 하네스: 전문가 에이전트 5종
│       ├── rfp-analyst.toml         (요구사항 전수 추출)
│       ├── domain-expert.toml       (보건의료 표준 지식 주입)
│       ├── solution-architect.toml  (기술 제안 설계)
│       ├── pm-expert.toml           (WBS·리스크·조건 대응)
│       └── compliance-reviewer.toml (전수성·확약 건전성 검증)
└── .agents/
    └── skills/                      ← 하네스: 스킬 3종
        ├── rfp-analysis/            (RFP 전수 추출 방법론)
        ├── medical-data-standards/  (도메인 지식 사전)
        └── proposal-writing/        (제안서 구조·문체 표준)
```

## 3. 트랙 A — 단일 프롬프트 실행

트랙 A는 **아무 자산도 없는 상태**가 기준선입니다. 그런데 이 디렉토리에서 `codex`를
실행하면 `AGENTS.md`의 하네스 규칙이 자동으로 로드되어 실험이 오염됩니다. 그래서
과업지시서만 빈 폴더로 복사해 깨끗한 조건을 만듭니다.

```bash
mkdir -p ../proposal-track-a
cp 과업지시서_데이터뱅크.hwpx ../proposal-track-a/
cd ../proposal-track-a
codex
```

`codex` 대화형 세션이 열리면 한 번에 요청합니다.

```
과업지시서_데이터뱅크.hwpx 를 읽고, 이를 기반으로 사업 제안서를
제안서A.docx 로 작성해 주세요. docx 생성은 Python(python-docx)으로 하고,
필요한 라이브러리는 pip3로 설치하세요.
```

Codex가 파일 압축 해제·라이브러리 설치·파일 생성 전에 **승인을 묻습니다**. 내용을
확인하고 허용하세요. (hwpx는 zip 컨테이너라, Codex가 스스로 압축을 풀어 본문을
읽어내는 과정을 지켜볼 수 있습니다.)

동봉된 `제안서A_단일패스.docx`가 이 방식으로 만든 예시 산출물입니다.
**일부러 못 쓴 것이 아닙니다** — 구조가 잡혀 있고 문장도 매끄러운, 단일 패스의
"괜찮은 평균"입니다. 이 기준선이 정직해야 비교가 의미를 갖습니다.

## 4. 트랙 B — 하네스 실행

다시 `proposal/` 디렉토리로 돌아와 `codex`를 실행하면 `AGENTS.md`가 자동으로
로드되고, `.codex/agents/`의 직무기술서와 `.agents/skills/`의 스킬을 이름으로
지목할 수 있습니다.

```bash
cd ../proposal
codex
```

대화형 세션에서 파이프라인 전체를 지시합니다.

```
과업지시서_데이터뱅크.hwpx 를 기반으로 사업 제안서 docx를 작성합니다.
.codex/agents/ 의 전문가 팀으로 다음 파이프라인을 실행하세요.

[1단계] rfp-analyst 에이전트를 생성해 요구사항 인벤토리
(artifacts/요구사항_인벤토리.md)를 작성하게 하세요.

[2단계] domain-expert 에이전트를 생성해 인벤토리에 도메인 지식을 주입하고
artifacts/표준매핑_전략노트.md 를 작성하게 하세요.

[3단계] solution-architect 와 pm-expert 에이전트를 병렬로 생성해 각각
기술 제안 장, 사업 관리·유지보수 장 원고를 artifacts/ 에 쓰게 하고,
둘 다 끝날 때까지 기다리세요.

[4단계] proposal-writing 스킬의 문서 구조에 맞춰 장별 원고를 하나로 통합하고,
Python(python-docx)으로 제안서B.docx 를 생성하세요. 필요한 라이브러리는
pip3로 설치하세요.

[5단계] compliance-reviewer 에이전트를 생성해 인벤토리 전건을 원고와 대조
검증하게 하세요. 누락·부실 판정이 나오면 해당 담당 에이전트에게 수정시키고,
전건 통과 판정이 나올 때까지 4~5단계를 반복하세요.
```

### 파이프라인이 실제로 하는 일

| 단계 | 에이전트 | 산출물 | 단일 프롬프트에는 없는 것 |
|---|---|---|---|
| 1 | rfp-analyst | `artifacts/요구사항_인벤토리.md` — REQ 11 / DOM 13 / CON 7 / DEL 4 전수 추출·ID 부여 | 목적 문단에 숨은 "가명화", 기대효과에 숨은 "보안 연계망", 표 각주의 강행 규정을 REQ로 승격 |
| 2 | domain-expert | 표준코드별 매핑 전략 노트 | FHIR KR Core ValueSet, KCD-7→8 전환표, 원내코드↔EDI, 개인정보보호법 §28-2 — **문서 밖 지식** |
| 3 | solution-architect ∥ pm-expert | 기술 제안 / WBS·리스크·조건 대응 (병렬) | 주 단위 WBS·임계경로, "미매핑률 ≤1%" 같은 검증 가능한 확약 |
| 4 | 부모 세션(통합) | `제안서B.docx` — 장별 원고를 proposal-writing 표준 구조로 통합 후 python-docx로 생성 | 장별로 나눠 쓴 원고를 표준 문서 구조에 맞춰 합치는 통합 단계 자체 (A는 한 번에 써서 이 단계가 없음) |
| 5 | compliance-reviewer | 컴플라이언스 검증표 | 인벤토리 ID 전건을 원고에서 대조 — 누락 발견 시 담당자 지목 수정 |

**관찰 포인트**:
- 서브에이전트끼리는 직접 대화하지 않습니다. 각자 산출물을 `artifacts/`에 파일로
  남기면, 부모(여러분과 대화 중인 Codex)가 파일 경로를 다음 단계에 전달하는
  **팀장 역할**을 합니다. 3단계의 병렬 집필이 가능한 것도 두 에이전트가 같은
  인벤토리를 기준표로 공유하기 때문입니다 (원리 3: 계약이 병렬성을 만든다).
- 5단계에서 compliance-reviewer는 원고를 직접 고치지 않고 담당 에이전트를
  지목합니다 — 만든 주체와 승인하는 주체가 분리되어 있습니다 (원리 4: 검증자 분리).

동봉된 `제안서B_하네스.docx`가 이 방식으로 만든 예시 산출물입니다.

## 5. 결과 비교 — 차이가 물성으로 남는 4개 지점

`제안서A_단일패스.docx`와 `제안서B_하네스.docx`를 나란히 열고 확인하세요.

| 비교 지점 | A (단일 패스) | B (하네스) |
|---|---|---|
| **배경지식** | "표준코드에 맞추어 매핑합니다" — 과업지시서 단어의 반복 | 6종 코드체계별 매핑 전략 표: KR Core ValueSet, KCD 전환표, 원내-EDI 매핑, ATC 2단계 (3.2절) |
| **전수 대응** | 눈에 띄는 요구 위주 — 보안 연계망·비정형 가명화 취급이 얕음 | REQ-01~11 추적 매트릭스, 검수 기준표로 재사용 명시 (2장) |
| **조건 조항** | 사실상 언급 없음 | CON-01~07 전건 수용 확약표: 3년 유지보수 SLA, 라이선스 검토서, 상주 5인, HW/SW 부담 (5장) |
| **검증 가능성** | "1단계 약 3주", "품질을 확보" | W1~W9 WBS + 임계경로, 미매핑률 ≤1%·가명화 누출 0건 + 이행 방안 병기 |

한 줄 요약: **A는 과업지시서를 요약한 제안서, B는 과업지시서에 응답한 제안서.**

## 6. 교육 포인트 — 차이는 어디서 왔는가

**① 차이의 근원은 문장이 아니라 중간 산출물.** B의 모든 우위는
`artifacts/요구사항_인벤토리.md` 하나에서 파생됩니다. 전수 추출·ID 부여라는
지루한 작업이 추적 매트릭스, 조건 확약표, 검수 기준표를 낳습니다. 하네스란
이 중간 산출물의 생산을 역할로 강제하는 장치입니다 (원리 1: What·Who·How 분리 —
직무기술서가 산출물을 강제합니다).

**② "배경지식 없음"의 해법은 지식의 사전화.** domain-expert가 똑똑해서가 아니라,
`medical-data-standards` 스킬에 지식이 사전으로 축적돼 있어서 가능합니다.
다음 의료 RFP에서는 이 사전이 그대로 재사용됩니다 — 이 과정 전체를 관통하는
"매핑 사전의 자산화" 원리의 제안서 버전입니다 (원리 2: 지식의 외부화).

**③ 검증자 분리가 누락을 잡는다.** 조건 조항 7건은 잘 쓰려는 노력으로 잡히지
않습니다. "인벤토리 ID를 하나씩 원고에서 찾아라"라는 별도 역할(compliance-reviewer)의
기계적 대조가 잡습니다. 잘 쓰는 것과 빠뜨리지 않는 것은 다른 능력입니다 (원리 4).

**④ A가 이기는 경우도 안다.** 내부 검토용 초안, 참고용 요약이라면 A의 속도가
정답입니다. 하네스는 "탈락 사유가 조항 하나의 누락일 수 있는" 고부담 문서에서
정당화됩니다 — 비용(과정의 복잡함)과 리스크(누락의 대가)를 저울질하는 것이
이 실험의 최종 교훈입니다.

## 7. 심화 과제

1. **평가위원 롤플레이**: 트랙 A용으로 만든 빈 폴더처럼 하네스 자산이 없는 별도
   폴더에 두 docx를 복사하고, 새 `codex` 세션에서 조달 평가위원 역할을 주어 A·B를
   채점시켜 보세요(요구 이해도/기술/관리/조건 대응 4개 항목). 점수 차이가 어느
   항목에서 벌어지는지 확인하세요.
2. **인벤토리 파괴 실험**: 요구사항 인벤토리에서 CON 섹션을 삭제하고 트랙 B의
   5단계를 재실행해 보세요 — compliance-reviewer가 원문 대조로 이를 잡아내는지
   (에이전트 정의의 에러 핸들링 조항)가 하네스 견고성의 시험대입니다.
3. **타 도메인 이식**: 다른 분야 과업지시서(예: 교육·물류)를 넣고, 무엇이 그대로
   동작하고(rfp-analysis, proposal-writing) 무엇을 교체해야 하는지
   (medical-data-standards → 해당 도메인 사전) 확인하세요. 스킬 분리 기준이
   왜 그렇게 설계됐는지 체감됩니다.
4. **한 줄 자동화**: 트랙 B의 파이프라인 지시 전체를
   `codex exec --sandbox workspace-write "…"` 한 줄에 담아 사람 개입 없이
   실행해 보세요. 사람이 없으므로 승인 대신 **샌드박스**가 안전장치가 됩니다 —
   `trend-report/` 실습의 정례 보고 자동화와 같은 원리입니다.
