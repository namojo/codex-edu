# 맛집 데이터로 배우는 스펙 → 하네스 개발 파이프라인

> 서울시청 인근 업무추진비 집행 데이터를 소재로, Codex 대화형 세션에서 **1단계: 프로젝트 스펙 문서**를 만들고
> **2단계: 그 스펙을 구현할 에이전트 팀(AGENTS.md + `.codex/agents/`)을 직접 설계·구현**하여
> "공무원 검증 맛집 지도" 웹앱을 완성하는 실습 교안입니다. 앞의 두 실습이 준비된 프롬프트를
> 따라 하는 과정이었다면, 이번에는 **설계도(스펙)와 조직(하네스)을 여러분이 직접 만드는 것**까지가 범위입니다.

## 1. 왜 이 실습인가

Codex에게 "맛집 지도 앱 만들어줘"라고 한 줄로 시키면, Codex는 수백 가지 결정(스택, 스키마, 색상, 랭킹 기준…)을 임의로 내립니다. 결과물은 그럴듯하지만 매번 다르고, 수정 요청이 반복됩니다.

이 실습은 그 문제를 **역할 분리**로 해결하는 훈련입니다.

| 계층 | 담당 | 산출물 |
|---|---|---|
| **무엇을 (What)** | 1단계에서 여러분이 Codex와 함께 작성 | `SEOUL_MATJIP_MAP_SPEC.md` — AI 코딩 에이전트가 해석 없이 따를 수 있는 계약서 |
| **누가 (Who)** | 2단계에서 Codex가 설계·생성 | `AGENTS.md` + `.codex/agents/*.toml` — 전문 에이전트 팀 정의 (파이프라인·생성-검증 패턴) |
| **어떻게 (How)** | 2단계에서 Codex가 설계·생성 | `.agents/skills/` — 도메인 규칙을 담은 스킬 (데이터 정제 규칙, 지오코딩 규칙 등) |

사람 조직에 비유하면, 스펙은 **발주 계약서**, `AGENTS.md`는 **팀 업무 수칙 게시판**, 에이전트 정의는 **직무기술서**, 스킬은 **업무 매뉴얼**입니다. 계약서 없이 조직부터 꾸리면 무엇을 만들지 모르고, 조직 없이 계약서만 있으면 아무도 일하지 않습니다 — 그래서 이 실습은 정확히 그 순서(What → Who/How)로 진행됩니다.

## 2. 데이터: 식당리스트_실습용.csv

서울시청 인근 업무추진비 집행 내역 형식의 실습용 데이터입니다. "공무원이 법인카드로 반복 방문한 식당 = 검증된 맛집"이라는 가설로 분석할 수 있습니다.

| 항목 | 값 |
|---|---|
| 집행 기록 | **15,144건** (2026년) |
| 고유 식당 | **2,946곳** |
| 총 집행액 | 약 20.3억 원 (건당 평균 약 13.4만 원) |
| 지역 분포 | 중구 11,848건(시청 인근 집중) · 종로구 1,154건 · 강남구 256건 … |
| 최다 방문 | 창고43시청점(146회), 구이구이(125회), 바닷가 수라간(120회) … |
| 컬럼 | 해당년도 · 해당월 · 집행일시 · 집행장소 · 집행금액 · 정리식당명 · 주소 |

### ⚠️ 의도된 함정 (데이터 정제 실습 포인트)

이 CSV는 실무 데이터의 지저분함을 그대로 담고 있습니다. 스펙과 스킬에 반드시 반영해야 할 이슈들입니다.

1. **인코딩**: UTF-8이 아니라 **CP949**, 구분자는 쉼표가 아니라 **탭(TSV)**, 줄바꿈은 CRLF — 그냥 `pd.read_csv()`하면 깨집니다.
2. **혼합 필드**: `집행장소` 컬럼에 상호와 주소가 함께 들어 있음 — 예: `명동칼국수(서울시 중구 세종대로 21길 48)`.
3. **명칭 불일치**: `주식회사 보원에프앤비` 같은 법인명 vs 실제 간판명, 같은 브랜드의 지점 표기 편차(`창고43시청점` vs `창고43 무교점`).
4. **좌표 부재**: 주소만 있고 위경도가 없음 — 지도에 올리려면 지오코딩 단계가 필요.

## 3. 실습 목표

**"시청뽑맛" — 공무원 검증 맛집 지도 웹앱**을 완성합니다.

- CSV를 정제하여 식당별 집계(방문 횟수, 총 집행액, 평균 단가, 최근 방문일) 생성
- 주소 지오코딩 후 지도(Leaflet 등) 위에 마커 + 방문 횟수 기반 랭킹 시각화
- 필터(구·월·금액대)와 식당 상세 패널을 갖춘 단일 페이지 웹앱


이 실습은 처음부터 끝까지 **대화형 세션 하나**에서 진행합니다. Codex가 파일 수정·명령 실행 전에 **승인을 묻습니다**. 내용을 확인하고 허용하세요 — 특히 2단계에서는 Codex가 어떤 파일을 만들려는지 승인 창에서 읽어 보는 것 자체가 학습입니다.

## 5. 1단계 — 프로젝트 스펙 문서 작성 (What)

먼저 Codex에게 코드를 시키는 게 아니라 **계약서를 쓰게** 합니다. `codex` 세션에 아래 프롬프트를 그대로 입력하세요.

```
"시청뽑맛" 웹앱의 프로젝트 스펙 문서를 SEOUL_MATJIP_MAP_SPEC.md 로 작성하세요.
이 단계의 산출물은 스펙 문서 하나뿐입니다 — 아직 코드를 작성하지 마세요.

## 앱 개요
- "시청뽑맛" — map 폴더의 식당리스트_실습용.csv(CP949 인코딩, 탭 구분, CRLF,
  15,144건)를 정제·집계하여 맛집 랭킹을 지도로 보여주는 웹앱
- 스택: React + Vite + TypeScript, 지도는 Leaflet, 백엔드 없음
  (빌드 타임에 CSV → JSON 전처리)
- 사용자: 시청 인근에서 점심 식당을 고르는 직장인
- 핵심 화면: 지도 뷰(마커 크기 = 방문 횟수), 랭킹 리스트, 식당 상세 패널,
  구/월/금액대 필터

## 스펙 작성 규칙 — "구체성이 곧 품질"
- 섹션 구성: overview / core_data_entities / screens / data_processing /
  scope_boundaries / success_criteria. 섹션마다 XML 태그로 구획하세요.
- 색상은 이름("파란색")이 아니라 hex 코드로, 크기는 "크게"가 아니라 픽셀값으로.
- core_data_entities 에는 Restaurant 엔티티의 모든 필드(name, address, lat, lng,
  visitCount, totalAmount, avgAmount, lastVisit …)를 타입과 함께 정의하세요.
- 데이터 함정 4종 — ① CP949+탭+CRLF 인코딩 ② 집행장소 컬럼의 상호·주소 혼합
  ③ 법인명-간판명 불일치와 지점 표기 편차 ④ 좌표 부재 — 의 처리 방침을
  data_processing 에 CRITICAL: 접두어로 명시하세요.
- scope_boundaries 에 이 프로젝트가 하지 않는 것(리뷰 작성, 실시간 데이터,
  로그인)을 명시하세요.
- 빈 상태(검색 결과 0건)와 에러 상태(지오코딩 실패 좌표)의 화면 동작을 정의하세요.
- success_criteria 는 숫자로 검증 가능하게 쓰세요
  (예: "2,946개 식당 중 좌표 확보율 95% 이상").
- 먼저 CSV의 첫 20행을 직접 읽어 실제 데이터 형태를 확인한 뒤 작성하세요.
```

**관찰 포인트**: Codex가 스펙을 쓰기 전에 CSV를 읽으며 인코딩·구분자를 스스로 확인하는지 보세요. 그리고 완성된 스펙이 프롬프트에는 없던 세부 결정(마커 색상, 랭킹 산식, 필터 동작)까지 못 박고 있는지 보세요 — 대화 속에서 휘발될 결정들이 파일로 굳는 순간입니다 (원리 2: 지식의 외부화).

### 좋은 스펙의 판별 기준 (교안 핵심)

스펙의 원칙은 **"구체성이 곧 품질"**입니다. 생성된 `SEOUL_MATJIP_MAP_SPEC.md`를 열어 다음을 직접 검사하세요. 미달 항목은 Codex에게 "이 항목을 보강하라"고 지시해 스펙을 고칩니다 — 코드가 아니라 **스펙을 고치는 습관**이 이 실습의 절반입니다.

- [ ] 색상이 이름("파란색")이 아니라 **hex 코드**(`#246BEB`)로, 크기가 "크게"가 아니라 **픽셀값**으로 적혔는가
- [ ] `core_data_entities`에 Restaurant 엔티티의 **모든 필드**(name, address, lat, lng, visitCount, totalAmount, avgAmount, lastVisit …)가 타입과 함께 정의됐는가
- [ ] 데이터 함정 4종의 처리 방침이 `CRITICAL:` 접두어로 명시됐는가 (예: `CRITICAL: CSV는 cp949 + 탭 구분으로 파싱한다`)
- [ ] `scope_boundaries`에 이 프로젝트가 **하지 않는 것**(리뷰 작성, 실시간 데이터, 로그인)이 적혔는가
- [ ] 빈 상태(검색 결과 0건), 에러 상태(지오코딩 실패 좌표)가 정의됐는가
- [ ] `success_criteria`가 숫자로 검증 가능한가 (예: "2,946개 식당 중 좌표 확보율 95% 이상")

이 스펙은 사람이 읽는 기획서가 아니라 **AI 코딩 에이전트가 해석 없이 실행하는 계약서**입니다. 모호함 하나가 곧 에이전트의 임의 결정 하나입니다.

## 6. 2단계 — 스펙을 구현할 에이전트 팀 설계·구현 (Who / How)

이제 계약서를 들고 **조직을 꾸립니다**. 이번에는 하네스를 남이 만들어 주는 게 아니라, Codex에게 하네스 구성 자체를 설계시키고 여러분이 그 설계를 검수합니다.

### 6-1. 하네스 설계·생성

같은 세션에 이어서 입력하세요.

```
SEOUL_MATJIP_MAP_SPEC.md 를 읽고, 이 스펙을 구현할 에이전트 팀(하네스)을
파일로 설계·생성하세요. 이번 지시의 산출물은 앱이 아니라 팀 그 자체입니다 —
아직 앱을 구현하지 마세요.

만들 파일:
1. AGENTS.md — 팀 공통 불변 규칙. 최소한 다음을 포함:
   - 정제는 표현을 바꾸지 수치를 바꾸지 않는다 (집행 건수·총액 보존)
   - 좌표를 얻지 못한 식당은 임의 좌표를 만들지 말고 [미확인]으로 보고한다
   - 스펙(SEOUL_MATJIP_MAP_SPEC.md)이 유일한 요구사항 소스다 —
     스펙에 없는 기능은 추가하지 않는다
   - 각 단계 산출물의 저장 위치 (data/, src/ 등)
2. .codex/agents/ 에 에이전트 정의 4종 (TOML — 핵심 필드는 name, description,
   developer_instructions):
   - data-cleaner  : CP949/탭 파싱, 혼합 필드 분리, 명칭 정규화,
                     식당별 집계 JSON 생성
   - geocoder      : 주소 → 위경도 변환, 실패 주소 목록화
   - app-builder   : 스펙을 유일한 요구사항 소스로 삼아 React 앱 구현
   - qa-validator  : 데이터 ↔ 화면 수치 교차 비교, success_criteria 검증.
                     검증 대상을 직접 수정하지 않는다
3. .agents/skills/ 에 도메인 스킬 최소 2종 (각 폴더에 SKILL.md):
   - matjip-data-rules : 정제 규칙과 명칭 통합 기준
   - geocoding         : 지오코딩 절차, 실패 처리, 캐싱 규칙

설계 규칙:
- 각 에이전트의 developer_instructions 에 입력 파일·산출물 경로·작업 기준을
  명시하세요. 에이전트끼리는 직접 대화하지 못하므로 파일이 곧 팀의 공용어입니다.
- 스킬 description 은 "언제 반드시 이 스킬을 쓰는지"가 드러나게 적극적으로 쓰세요.
- SKILL.md 본문은 간결하게 유지하고, 긴 상세 규칙(예: 명칭 통합 사전)은
  references/ 하위 파일로 분리한 뒤 본문에는 "언제 읽으라"는 포인터만 남기세요.
- 다 만들면 팀 구조를 표로 요약해 보고하세요.
```

기대되는 팀 구조:

```
[data-cleaner] ──▶ [geocoder] ──▶ [app-builder] ──▶ [qa-validator]
  CSV 정제·집계      주소→좌표 변환      스펙 기반 구현        경계면 교차 검증
```

| 에이전트 | 핵심 역할 | 전용 스킬 (예) |
|---|---|---|
| `data-cleaner` | CP949/탭 파싱, 혼합 필드 분리, 명칭 정규화, 식당별 집계 JSON 생성 | `matjip-data-rules` — 정제 규칙과 명칭 통합 기준 |
| `geocoder` | 주소 → 위경도 변환, 실패 주소 목록화 | `geocoding` — 지오코딩 절차·실패 처리·캐싱 |
| `app-builder` | 스펙을 유일한 요구사항 소스로 삼아 React 앱 구현 | (스펙 자체가 계약) |
| `qa-validator` | 데이터 ↔ 화면 수치 교차 비교, success_criteria 검증 | `qa-checklist` |

**관찰 포인트**: 생성된 `.codex/agents/*.toml`을 직접 열어 보세요. TOML의 핵심 3필드는 name·description·developer_instructions이며, 실전 자산에는 스킬 연결 등 확장 필드가 붙을 수 있습니다(예: `trend-report/.codex/agents/report-writer.toml`의 `[[skills.config]]`). `developer_instructions`가 "데이터를 잘 정제하라" 같은 덕담인지, 아니면 입력 경로·산출물 경로·판단 기준이 박힌 **직무기술서**인지가 이 하네스의 품질을 결정합니다 (원리 3: 계약이 병렬성·반복성을 만든다). 그리고 qa-validator의 정의에 "검증 대상을 직접 수정하지 않는다"가 들어 있는지 확인하세요 (원리 4: 검증자 분리).

### 하네스 설계 검수 기준 (교안 핵심)

- **에이전트 정의는 반드시 파일로**: `.codex/agents/{name}.toml` 없이 프롬프트에 역할을 직접 넣는 것은 금지 — 다음 세션 재사용과 팀 통신 프로토콜 보장이 하네스의 존재 이유입니다.
- **QA는 "존재 확인"이 아니라 "경계면 교차 비교"**: 정제된 JSON의 `visitCount` 합계와 화면 랭킹의 수치를 **동시에 읽고 비교**해야 QA입니다. "파일이 있네요"는 QA가 아닙니다. 또한 전체 완성 후 1회가 아니라 **모듈 완성 직후 점진 실행**(incremental QA)이 원칙입니다.
- **스킬 description은 적극적으로**: "데이터를 정제하는 스킬"(나쁨) 대신 "식당리스트 CSV의 CP949 파싱, 혼합 필드 분리, 법인명-상호 통합, 식당별 집계를 수행. 이 CSV를 다루는 모든 작업에서 반드시 사용할 것"(좋음). description이 스킬의 유일한 트리거입니다.
- **본문은 얇게, 상세는 references/로**: SKILL.md 본문은 짧고 실행 가능하게, 상세 규칙(예: 명칭 통합 사전)은 `references/`로 분리하고 "언제 읽으라"는 포인터만 남깁니다. 컨텍스트 윈도우는 공공재입니다.

미달 항목이 있으면 Codex에게 해당 파일만 고치게 하세요 — 하네스도 스펙처럼 **파일을 고쳐서** 개선합니다.

### 6-2. 팀 실행

설계가 검수를 통과했으면, 같은 세션에서 팀을 가동합니다.

```
이제 방금 만든 팀으로 SEOUL_MATJIP_MAP_SPEC.md 를 구현하세요.

[1단계] data-cleaner 에이전트를 생성해 matjip-data-rules 스킬에 따라
CSV를 정제·집계하고 data/restaurants.json 을 만들게 하세요.
정제 전후의 집행 건수와 총액이 보존됐는지 근거와 함께 보고받으세요.

[2단계] geocoder 에이전트를 생성해 주소를 위경도로 변환하게 하세요.
실패한 주소는 [미확인] 목록으로 보고받으세요.

[3단계] app-builder 에이전트를 생성해 스펙대로 React 앱을 구현하게 하세요.

[4단계] qa-validator 에이전트를 생성해 데이터와 화면 수치를 교차 비교하고
스펙의 success_criteria 를 검증하게 하세요.
문제 항목은 app-builder 에게 수정시키고, 재검증까지 마치세요.

각 단계가 끝날 때마다 산출물 요약을 보고한 뒤 다음 단계로 넘어가세요.
```

⚠️ 지오코딩은 외부 API 호출이 필요할 수 있습니다. Codex가 네트워크 접근이나 API 키를 요구하는 명령의 승인을 물으면 내용을 확인하고 판단하세요. API를 쓸 수 없는 환경이라면 그 자체가 관찰 기회입니다 — 좌표를 얻지 못한 식당이 임의 좌표가 아니라 `[미확인]`으로 보고되는지(AGENTS.md의 불변 규칙이 작동하는지) 지켜보세요.

## 7. 검수 — success_criteria 기준

팀 실행이 끝나면 스펙의 `success_criteria`를 기준으로 검수합니다.

- [ ] 정제 결과: 고유 식당 수가 명칭 통합 후에도 근거와 함께 보고되는가 (원본 2,946 → 통합 후 N, 통합 규칙 로그 첨부)
- [ ] 수치 보존: 정제 전후 집행 건수(15,144)와 총액(≈20.3억)이 일치하는가 — **정제는 표현을 바꾸지 수치를 바꾸지 않는다**
- [ ] 지오코딩: 좌표 확보율과 실패 주소 목록이 `[미확인]`으로 보고되는가 (임의 좌표 생성 금지)
- [ ] 화면-데이터 일치: 랭킹 1위 식당의 방문 횟수가 JSON 집계값과 동일한가
- [ ] 스펙 준수: 스펙에 없는 기능이 추가되지 않았는가 (scope_boundaries 위반 여부)

## 8. 교육 포인트

**① 스펙은 프롬프트의 상위 호환이다.** 프롬프트는 대화 속에서 휘발되지만, 스펙은 파일로 남아 어떤 에이전트가 몇 번을 실행해도 같은 요구사항을 전달합니다. "AI에게 시키는 일이 반복된다면, 프롬프트를 다듬지 말고 스펙을 만들어라"가 이 실습의 첫 번째 교훈입니다.

**② What/Who/How의 분리가 재사용을 만든다.** 스펙(What)을 바꾸면 같은 하네스로 다른 앱을 만들 수 있고, 하네스(Who/How)를 바꾸면 같은 스펙을 다른 팀 구조로 구현해 비교할 수 있습니다. 셋이 한 프롬프트에 뭉쳐 있으면 아무것도 재사용할 수 없습니다 (원리 1).

**③ 더러운 데이터가 좋은 교재다.** CP949 인코딩, 혼합 필드, 명칭 불일치는 버그가 아니라 현실입니다. 이런 함정의 처리 방침이 스펙에 `CRITICAL:`로 명시되어 있는지, 정제 규칙이 스킬로 외부화되어 있는지가 "장난감 데모"와 "실무 파이프라인"을 가릅니다 (원리 2).

**④ 모호함의 총량 보존 법칙.** 스펙에서 정의하지 않은 것은 에이전트가 임의로 정합니다. 모호함은 사라지지 않고 구현 단계의 무작위성으로 이전될 뿐입니다. 스펙 작성에 쓰는 30분이 구현 재작업 3시간을 줄입니다.

## 9. 심화 과제

1. **스펙 A/B 실험**: 같은 요구를 ① 한 줄 프롬프트로, ② 완성된 스펙으로 각각 구현시키고 결과의 일관성·완성도·재작업 횟수를 비교해 보세요. 스펙의 가치가 체감됩니다.
2. **명칭 통합 Human-in-the-loop**: `data-cleaner`가 확신 없는 명칭 통합 후보(예: 창고43시청점 ↔ 창고43 무교점 — 같은 브랜드, 다른 지점)를 사람에게 승인받아 스킬의 `references/` 사전에 축적하는 루프를 설계해 보세요.
3. **보고서 에이전트 증원**: `report-writer.toml`을 직접 작성해 팀에 추가하고, 정제·집계 결과로 "이달의 시청 맛집 리포트"(마크다운 뉴스레터)를 자동 생성시켜 보세요 — `trend-report/` 실습의 조사→집필 분업 구조(조사 노트를 재료로 집필 전담 에이전트가 종합)를 이 팀 위에서 재현하는 연습입니다.
4. **스펙 리버스 엔지니어링**: 완성된 앱만 보고 스펙을 역으로 작성해 원본 스펙과 비교해 보세요 — 스펙의 어떤 섹션이 구현에서 소실되기 쉬운지 발견할 수 있습니다.
5. **정례화**: 데이터가 매달 갱신된다고 가정하고, 팀 실행 지시를 `codex exec --sandbox workspace-write "…"` 한 줄로 옮겨 보세요. 대화형에서 승인이 하던 안전장치를 자동화에서는 **샌드박스**가 대신한다는 것을 확인할 수 있습니다.
